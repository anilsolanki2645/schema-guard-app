🧩 Schema Guard Troubleshooting Guide

Find details, explanation, and resolution procedures for common pipeline schema-guard failures.

❌ Exit Code 1: Schema Drift Detected Violations ▼

Why it happens: The database schema has shifted away from the design specifications in the contract baseline. This is usually due to a column deletion, a change in nullability constraint (e.g. adding NOT NULL), or a mismatch in data types.

Resolution Options:

  1. Revert the database schema change if the drift was accidental or unauthorized.
  2. If the change was intentional, update the contract file and run schema-guard snap --force to capture the new baseline schema.
  3. If this is an acceptable data type upgrade (e.g. increasing character size), define an allowed_drift rule inside your contract.
❌ Exit Code 3: Snapshot Integrity Check FAILED ▼

Why it happens: The baseline snapshot file (e.g., snapshots/orders.json) was modified manually, corrupted, or has an invalid checksum hash.

Resolution Options:

  1. Avoid editing snapshots manually. Always use the CLI tool to update snapshots.
  2. To fix a corrupted snapshot, run schema-guard snap --force --contract <contract.yaml> from a trusted database environment to regenerate a fresh integrity-verified baseline.
❌ Exit Code 4: Contract Signature Lock Failed ▼

Why it happens: The signature hash of the yaml contract configuration doesn't match the verification key stored in the lock file sidecar (e.g. contracts/orders.yaml.lock). This occurs when someone edits the yaml contract design without rehashing and signature locking it.

Resolution Options:

  1. Always lock the design parameters after making changes using schema-guard lock --contract <contract.yaml>.
  2. In the web UI, you can click Update Contract YAML under AI resolution panel suggestions to auto-sign the configuration file.
  3. In the YAML Contract Builder tab, you can click Lock Contract Signature to rehash the specifications before deployment.
